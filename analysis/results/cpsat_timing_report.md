# rot4-NTIL 计时表征报告（CP-SAT R9b 快速模型）

**日期**：2026-07-13
**目的**：用户指令「结合当前所有理论去从小 m 开始，看看时间」——量化求解耗时随 m 的增长。
**方法**：CP-SAT + R9b 快速 2-因子模型（`solve_m37_r9b.py`）。该模型编码了**所有当前理论**：
- R9b 2-因子结构 `rowSum[i]+colSum[i]==2`（Th-44），
- 逐线加权 at-most-2（`line_cons`，即 R8/SIRH Part III 的二次完备 CSP）。
快速生成器 `reps→lifted points→directions` 把模型构建从旧版的 O(m⁶) 降到 O(m²·|G|·|D|)（m=37 实测构建 262s）。

## 2. 关键发现：难度在 m≈17 处剧变

- m≤16：CP-SAT **秒级**找到 OPTIMAL 解，且 `verify_cells` 真值校验全部 `no-3-collinear=True`。
- m≥17：120s 内**无法**找到解（UNKNOWN）。
- 求解时间从 m=16 的 5.4s 跳到 m=17 的 >120s（**≥20× 跃迁**），且 m=17→18 之间未出现缓和，说明这是一个**相变式难度边界**，而非平滑增长。
- 拟合：m=14→16 约每步 ×2–×2.7，符合组合搜索空间的指数膨胀；到 m=17 超出 120s 窗口。

## 3. 与「旧墙 m=20」的关系（重要澄清）

- **旧墙（m=20，UNKNOWN@600s）是「构建墙」，不是「求解墙」**：旧版 `generate_constraints` 按 `directions→points→reps` = O(m⁶) 循环，m=37 时卡在构建阶段（0 字节日志），**从未真正进入求解器**。
- 用 R9b 快速生成器后，构建不再卡死（m=37 实测 262s 完成），**真正的求解器难度边界移到 m≈17**。
- **m=3..36 全部已知有 rot4 解**（存在性已由先前 R9b 扫掠/已知解文件证明）。因此 m≥17 的 UNKNOWN@120s 是**纯求解器速度边界**，**不等于不存在解**。m=37 是第一个存在性 OPEN 的 case。

## 5. C++ 构造核（csearch.cpp）状态

- 本会话修复了关键正确性 bug：`build_map` 原先按「点对」计数再算 C(gmap,3)，与增量逻辑按「点数」算 C(t,3) 量纲不一致 → 假阳性（cur 误到 0 而真值 verify_bad=36）。改为 `line→{point indices}` 集合精确统计 entries/line，与增量一致；并加退化线守卫 + FOUND 必经 `verify_bad==0` 真值校验。
- 直线 key 打包为 64-bit uint64，退火加 reheat 防卡 best=4 局部极小。
- **速度结论**：逻辑已对（m=10 restart0 收敛到 best=4，证明有效），但仅 **~10⁴ moves/sec**（per-move O(64m) line_of+map），m=10 需分钟级，m=37 估计数小时。
- **按用户纪律「比旧方法慢就关」**：CP-SAT 是更快的旧方法且正确，故 C++ 核降为 **m=37 备选构造路线**（仅当 CP-SAT 在 37 前撞墙才启用）。脚本保留：`csearch.cpp`、`verify_cells.py`（独立暴力校验）、`sweep_csearch.sh`。

## 6. 结论与建议

- 「看时间」答案：**CP-SAT R9b 在 m≤16 秒级，m≈17 起进入分钟级以上求解器速度墙**；m=37 远在此边界之外。
- 计算路线对 m=37 已实质封顶：CP-SAT 不现实（即便长攻击也大概率 UNKNOWN），C++ 核太慢。
- **m=37 的突破更可能来自结构性定理**（R9g 指向 T12 环型细化 / 非代数 2-因子构造），而非更多算力。长 CP-SAT 攻击作为受权的最后一搏在后台运行，结果将自动汇报。
